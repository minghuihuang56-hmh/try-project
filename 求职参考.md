**酒店预订数据分析 Agent** | Streamlit · DeepSeek API · Function Calling · Pandas · Plotly

## 二、面试高频问题

### 1. Agent 和普通 API 调用有什么区别？

- **普通 API 调用**：单次请求 → 单次回复，无工具执行能力，LLM 只能凭训练知识回答
- **Agent 模式**：LLM 自主决策 → 调用外部工具（查询数据库、执行代码、画图）→ 获取结果 → 再交给 LLM 解读，形成"思考→行动→观察"循环
- **核心区别**：Agent 有**工具使用能力**和**环境反馈闭环**，能获取实时数据并执行计算，回答是有数据支撑的
- **本项目体现**：LLM 不直接回答"取消率前5国家"，而是先生成 SQL 查数据，执行后把结果表格发给 LLM，LLM 再写出"葡萄牙取消率最高达 42%"这样的结论

### 2. Function Calling 的原理是什么？

- **本质**：在 API 请求中附上工具的描述（JSON Schema），包括函数名、参数列表、参数类型、用途说明
- **LLM 判断流程**：LLM 理解用户意图 → 匹配最合适的工具 → 生成结构化的参数 JSON → 返回 tool_use 标记（非自然语言）
- **开发者流程**：检测到 tool_use → 解析参数 → 执行本地函数 → 将结果包装为 tool_result 消息发回 → LLM 据此生成最终回复
- **关键点**：工具描述（description）要写得足够清晰才能让 LLM 准确选择；参数 schema 要用 additionalProperties: false 防止幻觉参数

### 3. 你的 run_python 安全沙箱怎么实现的，有什么局限？

**实现：**
- 自定义 `__builtins__`：只保留纯计算函数（abs, len, range, max 等），移除 open/eval/exec/__import__
- 自定义 `__import__` 钩子：白名单只允许 pandas/numpy/plotly/scipy/math 等科学计算库，拦截 os/sys/subprocess
- 注入预置变量：pd, np, px, go 直接放入命名空间，即使代码不 import 也能用
- stdout 重定向：捕获 print 输出到 StringIO，不影响主程序

**局限：**
- 不是真正的沙箱隔离，Python 的 exec 本身不是安全边界，有经验的攻击者可能用 `ctypes` 或对象协议绕过
- 生产环境应该用 Pyodide / Docker 容器 / gVisor 做进程级隔离
- 本项目的定位是本地/个人使用，不涉及多租户，当前实现足以防御用户无意误操作

### 4. 多轮对话的上下文是怎么管理的？

- **前端**：用 Streamlit 的 `st.session_state.messages` 存对话历史，每条消息格式为 `{"role": "user"|"assistant", "content": str, "fig": Figure|None}`
- **API 发送前**：剥离 fig 等自定义字段，只传 `role` + `content` 给 DeepSeek API（避免 API 不认识自定义字段报错）
- **系统提示词**：每次请求在 system prompt 中注入数据集 schema + 前 5 行样本数据，让 LLM 理解当前数据结构
- **上下文窗口**：没有主动截断（当前 11 万条数据 + 几轮对话远未达到 DeepSeek 的上下文限制），生产环境需要按 token 数滑动窗口或摘要历史

### 5. 如果让你把这个项目扩展成支持多数据源，你会怎么设计？

**分层抽象（借鉴适配器模式）：**

```
                 ┌──────────────────────────┐
                 │   DataSource 抽象接口      │
                 │   - get_schema()          │
                 │   - query(sql)            │
                 │   - get_sample(n)         │
                 └──────────┬───────────────┘
            ┌───────────────┼───────────────────┐
            │               │                   │
    ┌───────▼──────┐ ┌──────▼──────┐ ┌────────▼────────┐
    │  CSVSource   │ │  SQLSource  │ │  APISource      │
    │  (当前实现)   │ │ (MySQL/PG) │ │ (REST/gRPC)    │
    └──────────────┘ └─────────────┘ └─────────────────┘
```

**具体设计：**
- 定义抽象基类 `DataSource`，统一接口：`get_schema()` 返回字段信息，`query(sql)` 返回 DataFrame，`get_sample(n)` 返回样本
- 每个数据源实现自己的适配器：CSVSource 用 pandas 读文件，SQLSource 用 SQLAlchemy 连数据库，APISource 把接口响应转为 DataFrame
- router.py 不感知具体数据源，只调用 DataSource 接口，tool definitions 里动态注入当前数据源的 schema 描述
- 新增数据源只需写一个新的适配器类，不需要改路由逻辑
- 扩展性好，未来还可以加：ExcelSource、ParquetSource、MongoSource、BigQuerySource

**额外升级点：**
- 加缓存层：相同查询 5 分钟内命中缓存，提速降本
- 加数据抽样策略：超大数据集自动抽样 → 分析 → 再在全量上验证，避免 OOM
- 加查询安全审计：对 SQL 查询做 AST 解析，拦截 DDL/DML 操作

## 三、面试追问方向

面试官可能会顺着你的项目问的扩展方向：

- **"你是怎么做 Function Calling 的 Error Handling 的？"**
  → 工具执行失败时 error message 回传给 LLM，LLM 判断是否重试或换工具；max_iterations=10 防止死循环；异常分三类（API 401 / timeout / rate limit）展示对应友好提示

- **"Prompt 注入怎么防御？"**
  → system prompt 限定了角色和工具范围；自定义 import 白名单防止恶意代码；用户输入只作为工具参数传递，不拼接进 SQL（SQL 由 LLM 生成并由 SQLite 执行，安全）

- **"多工具串联时数据怎么传递的？"**
  → 比如"先查每月取消率再画图"：第一步 run_python 计算结果存入 result，第二步 make_chart 直接读 df（数据一直在内存中），中间结果通过 tool_result message 回传给 LLM 做上下文

- **"为什么选 DeepSeek 而不是直接用 GPT？"**
  → 成本优势（DeepSeek 约为 GPT-4o 的 1/10），中文能力出色，OpenAI 兼容格式切换成本低
